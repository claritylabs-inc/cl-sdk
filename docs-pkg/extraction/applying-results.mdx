The extraction pipeline returns a fully typed `ExtractionResult` containing the structured document, retrieval-ready chunks, cumulative token usage, and usage-reporting metadata. No post-processing or normalization step is needed — results are ready for persistence and querying.

## ExtractionResult

```typescript
interface ExtractionResult {
  document: InsuranceDocument;  // Fully typed policy or quote
  chunks: DocumentChunk[];       // Ready for vector storage
  tokenUsage: TokenUsage;        // Cumulative { inputTokens, outputTokens }
  usageReporting: {
    modelCalls: number;
    callsWithUsage: number;
    callsMissingUsage: number;
  };
}
```

```typescript
const { document, chunks, tokenUsage, usageReporting } = await extractor.extract(pdfBase64, "doc-123");
```

## InsuranceDocument

`InsuranceDocument` is a discriminated union of `PolicyDocument` and `QuoteDocument`, distinguished by the `type` field:

```typescript
type InsuranceDocument = PolicyDocument | QuoteDocument;
```

Both types share a large set of base fields and diverge on type-specific fields.

### Type narrowing

Use the `type` discriminator to narrow the union:

```typescript
const { document } = await extractor.extract(pdfBase64);

if (document.type === "policy") {
  console.log(document.policyNumber);     // string
  console.log(document.effectiveDate);    // string
} else {
  console.log(document.quoteNumber);      // string
  console.log(document.subjectivities);   // Subjectivity[] | undefined
}
```

## DocumentChunk

The pipeline's `chunks` output contains retrieval-ready chunks produced by `chunkDocument()`. Each chunk is designed for vector storage and semantic search.

```typescript
interface DocumentChunk {
  /** Deterministic ID: `${documentId}:${type}:${index}` */
  id: string;
  /** Source document ID */
  documentId: string;
  /** Chunk type for filtering */
  type: "carrier_info" | "named_insured" | "coverage" | "endorsement"
      | "exclusion" | "condition" | "section" | "declaration"
      | "loss_history" | "premium" | "supplementary";
  /** Human-readable text for embedding */
  text: string;
  /** Structured metadata for filtering */
  metadata: Record<string, string>;
}
```

### Chunk types

| Type | Source | Content |
|------|--------|---------|
| `carrier_info` | Carrier extractor | Carrier name, NAIC, AM Best, MGA, underwriter, broker |
| `named_insured` | Named insured extractor | Insured name, DBA, address, entity type |
| `coverage` | Coverage limits extractor | One chunk per coverage with limits and deductibles |
| `endorsement` | Endorsements extractor | One chunk per endorsement with form number and content |
| `exclusion` | Exclusions extractor | One chunk per exclusion |
| `condition` | Conditions extractor | One chunk per policy condition |
| `section` | Sections extractor | Raw section content |
| `declaration` | Declarations extractor | Line-specific declaration data |
| `loss_history` | Loss history extractor | Loss runs and claim records |
| `premium` | Premium breakdown extractor | Premium, taxes, fees, payment plans |
| `supplementary` | Supplementary extractor | Regulatory context, TPA, contacts |

### Deterministic chunk IDs

Chunk IDs follow the pattern `${documentId}:${type}:${index}`, where `index` is a zero-based counter within each type. This makes chunk IDs deterministic and idempotent — re-extracting the same document produces the same chunk IDs, enabling safe upserts.

```typescript
// Example chunk IDs for document "doc-123":
// "doc-123:carrier_info:0"
// "doc-123:coverage:0"
// "doc-123:coverage:1"
// "doc-123:coverage:2"
// "doc-123:endorsement:0"
// "doc-123:endorsement:1"
```

## Persisting results

The `ExtractionResult` is designed to feed directly into the SDK's storage interfaces:

```typescript
import { createExtractor } from "@claritylabs/cl-sdk";

const extractor = createExtractor({
  generateText: myGenerateText,
  generateObject: myGenerateObject,
});

const { document, chunks, tokenUsage, usageReporting } = await extractor.extract(pdfBase64, "doc-123");

// Save structured document to DocumentStore
await documentStore.save(document);

// Save chunks to MemoryStore for vector search
await memoryStore.addChunks(chunks);
```

<Callout type="info">
`DocumentStore` and `MemoryStore` are interfaces defined by the SDK. The SDK includes a reference SQLite implementation, but you can implement these interfaces with any storage backend.
</Callout>

## Token usage

The `tokenUsage` field on `ExtractionResult` provides cumulative input and output token counts across all model calls that actually return usage metadata.

```typescript
interface TokenUsage {
  inputTokens: number;
  outputTokens: number;
}
```

```typescript
const { tokenUsage, usageReporting } = await extractor.extract(pdfBase64);
console.log(`Cost: ${tokenUsage.inputTokens} input + ${tokenUsage.outputTokens} output tokens`);
console.log(`Usage reported for ${usageReporting.callsWithUsage}/${usageReporting.modelCalls} model calls`);
```

For per-call tracking, use the `onTokenUsage` callback in `ExtractorConfig`. If some provider calls omit usage, `usageReporting.callsMissingUsage` will make that visible.

## Null handling

The assembler uses `sanitizeNulls` internally to convert `null` values to `undefined` throughout the document. This ensures compatibility with frameworks like Convex that reject `null` for optional fields. You do not need to call `sanitizeNulls` yourself on extraction results.
