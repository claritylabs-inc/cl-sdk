CL SDK defines storage interfaces that power the query agent, application processing, source grounding, and extraction pipelines. All interfaces are provider-agnostic -- you implement them against your own database.

## DocumentStore

CRUD for `InsuranceDocument` objects produced by the extraction pipeline.

```typescript
interface DocumentStore {
  save(doc: InsuranceDocument): Promise<void>;
  get(id: string): Promise<InsuranceDocument | null>;
  query(filters: DocumentFilters): Promise<InsuranceDocument[]>;
  delete(id: string): Promise<void>;
}
```

The `query` method accepts optional filters to narrow results:

```typescript
interface DocumentFilters {
  type?: "policy" | "quote";
  carrier?: string;
  insuredName?: string;
  policyNumber?: string;
  quoteNumber?: string;
}
```

## MemoryStore

Vector search over document chunks and conversation history. This is the primary retrieval layer for the query agent.

```typescript
interface MemoryStore {
  addChunks(chunks: DocumentChunk[]): Promise<void>;
  search(query: string, options?: { limit?: number; filter?: ChunkFilter }): Promise<DocumentChunk[]>;
  addTurn(turn: ConversationTurn): Promise<void>;
  getHistory(conversationId: string, options?: { limit?: number }): Promise<ConversationTurn[]>;
  searchHistory(query: string, conversationId?: string): Promise<ConversationTurn[]>;
}
```

Chunk searches can be narrowed with a `ChunkFilter`:

```typescript
interface ChunkFilter {
  documentId?: string;
  type?: DocumentChunk["type"];
  metadata?: Record<string, string>;
}
```

`search` performs semantic (vector) search over chunk text. Your implementation is responsible for embedding the query and computing similarity — the SDK never imports an embedding provider directly.

## ApplicationStore

Persistent state for the application processing pipeline. Each application moves through statuses (`classifying`, `extracting`, `filling`, `collecting`, `confirming`, `mapped`) and the store persists the current state between user interactions.

```typescript
interface ApplicationStore {
  save(state: ApplicationState): Promise<void>;
  get(id: string): Promise<ApplicationState | null>;
  list(filters?: ApplicationListFilters): Promise<ApplicationState[]>;
  delete(id: string): Promise<void>;
}

interface ApplicationListFilters {
  status?: ApplicationState["status"];
  title?: string; // fuzzy match
}
```

## BackfillProvider

Vector-based answer reuse for application processing. When a new application is processed, the backfill provider searches previously answered fields to pre-fill matching questions.

```typescript
interface BackfillProvider {
  searchPriorAnswers(
    fields: { id: string; label: string; section: string; fieldType: string }[],
    options?: { limit?: number },
  ): Promise<PriorAnswer[]>;
}

interface PriorAnswer {
  fieldId: string;
  value: string;
  source: string;
  relevance: number;
}
```

The provider uses vector similarity over field labels and section context to find answers from prior applications, extracted policies, and business context.

## SourceStore

Source grounding persists raw evidence separately from vector chunks. `SourceStore` extends `SourceRetriever`, so the same implementation can store spans and serve quoted-evidence retrieval.

```typescript
interface SourceStore extends SourceRetriever {
  addSourceSpans(spans: SourceSpan[]): Promise<void>;
  addSourceChunks(chunks: SourceChunk[]): Promise<void>;
  getSourceSpan(id: string): Promise<SourceSpan | null>;
  getSourceSpansByDocument(documentId: string): Promise<SourceSpan[]>;
  getSourceChunksByDocument(documentId: string): Promise<SourceChunk[]>;
  deleteDocumentSource(documentId: string): Promise<void>;
}
```

The SDK includes `MemorySourceStore` for tests and local development. Production hosts usually back this interface with a document database plus lexical/vector indexes.

## How the interfaces connect

The four interfaces form a data flow through the SDK's pipelines:

1. **Extraction** produces an `InsuranceDocument`, `DocumentChunk[]`, and optional `SourceSpan[]`/`SourceChunk[]` -- save the document to `DocumentStore`, add chunks to `MemoryStore`, and persist source evidence to `SourceStore`
2. **Query agent** reads from document, memory, and optional source stores -- `DocumentStore` for structured lookup, `MemoryStore` for semantic chunks/history, and `SourceRetriever` for source-span evidence
3. **Application pipeline** uses `ApplicationStore` for persistent state across the multi-step processing flow, and `BackfillProvider` to pre-fill fields from prior answers
4. **PCE and case workflows** use `SourceRetriever` and quoted evidence validation to build reviewable packets
5. **Conversation turns** stored in `MemoryStore` enable cross-conversation memory -- the query agent can reference earlier interactions

```
Extraction Pipeline
    ├─► InsuranceDocument ──► DocumentStore
    ├─► DocumentChunk[]   ──► MemoryStore
    └─► SourceSpan[]      ──► SourceStore
                                  │
Query Agent ◄─────────────────────┤
    │                             │
    └─► ConversationTurn ─────────┘

Application Pipeline
    ├─► ApplicationState ──► ApplicationStore
    └─► BackfillProvider (reads prior answers)

PCE / Case Workflows
    └─► SourceRetriever (quoted evidence)
```

## SQLite reference implementation

CL SDK ships a reference SQLite implementation of `DocumentStore` and `MemoryStore` using `better-sqlite3`:

```typescript
import { createSqliteStore } from "@claritylabs/cl-sdk";

const store = createSqliteStore({
  path: "./data/store.db",
  embed: async (text) => {
    // Your embedding function — returns number[]
    return await myEmbeddingProvider.embed(text);
  },
});

// store.documents: DocumentStore
// store.memory: MemoryStore
// store.close(): void
```

<Callout type="info">
  The SQLite implementation is a reference for development and testing. For production use, implement the interfaces against your own database (Postgres with pgvector, Pinecone, Convex, etc.).
</Callout>

The `embed` callback follows the same provider-agnostic pattern as the rest of CL SDK — you wrap your preferred embedding provider (OpenAI, Anthropic, Cohere, etc.) into a plain function `(text: string) => Promise<number[]>`.
