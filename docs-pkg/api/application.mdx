## Pipeline

### `createApplicationPipeline(config)`

Create an application processing pipeline with persistent state, bounded workflow planning, optional auto-fill, and vector-based backfill.

```typescript
import { createApplicationPipeline } from "@claritylabs/cl-sdk/application";

const pipeline = createApplicationPipeline({
  generateText, generateObject,
  applicationStore, documentStore, memoryStore,
  orgContext: [{ key: "company_name", value: "Acme", category: "company_info" }],
  concurrency: 4,
});
```

**Returns:** `{ processApplication, processReply, generateCurrentBatchEmail, getConfirmationSummary, createApplicationRun, planNextQuestions, proposeContextWrites, buildApplicationPacket }`

### `pipeline.processApplication(input)`

Process a new PDF: classify -> extract -> plan optional backfill/auto-fill/search -> batch if needed.

| Parameter | Type | Description |
|-----------|------|-------------|
| `input.pdfBase64` | `string` | Base64-encoded application PDF |
| `input.sourceSpans` | `SourceSpan[]?` | Optional page/field evidence for the application PDF |
| `input.applicationId` | `string?` | Application ID |
| `input.template` | `ApplicationTemplate?` | Template snapshot to pin on the run |
| `input.questionGraph` | `ApplicationQuestionGraph?` | Pre-authored or extracted graph |
| `input.context` | `AgentContext?` | Platform context |

**Returns:** `Promise<{ state: ApplicationState; tokenUsage: TokenUsage; reviewReport: ApplicationQualityReport }>`

### `pipeline.processReply(input)`

Process a user reply: classify intent -> plan actions -> parse answers / handle lookups / explain fields -> advance batch.

| Parameter | Type | Description |
|-----------|------|-------------|
| `input.applicationId` | `string` | Application ID |
| `input.replyText` | `string` | User's reply |
| `input.replySourceSpanIds` | `string[]?` | Stable source span IDs for the reply or attachments |
| `input.context` | `AgentContext?` | Platform context |

**Returns:** `Promise<{ state, intent, fieldsFilled, responseText?, tokenUsage, reviewReport }>`

Application workflow gates optional work from the current state. Backfill requires an available backfill provider and unfilled fields, context auto-fill requires relevant org context, document search targets selected high-value fields, and reply handling runs only the agents needed for the classified intent.

Source grounding is optional but recommended. `sourceSpans` are forwarded to classification and field extraction. `replySourceSpanIds` preserve provenance for user-supplied answers, while backfill providers and memory chunks can attach source spans to auto-filled values.

### `pipeline.generateCurrentBatchEmail(applicationId, opts?)`

Generate the email for the current question batch. **Returns:** `Promise<{ text, tokenUsage }>`

### `pipeline.getConfirmationSummary(applicationId)`

Generate a formatted summary for user review. **Returns:** `Promise<{ text, tokenUsage }>`

### `pipeline.createApplicationRun(input)`

Create a persistent run from a pinned `ApplicationTemplate` without reprocessing a PDF. The run stores `templateId`, `templateVersion`, `templateSnapshot`, `questionGraph`, and flattened fields.

### `pipeline.planNextQuestions(applicationId, limit?)`

Return active unanswered fields for the current run. Inactive conditional fields are skipped until their parent answers make them active.

### `pipeline.proposeContextWrites(applicationId)`

Return reusable org-context proposals from confirmed/high-confidence application answers and persist them on state as `contextProposals`.

### `pipeline.buildApplicationPacket(input)`

Build and persist an `ApplicationPacket` with source-backed answers, missing required field IDs, quality report, and optional submission notes. The run status becomes `packet_ready` only when packet validation has no blocking issues.

## Graph and packet helpers

### `extractQuestionGraphFromFields(fields, options)`

Build a versioned `ApplicationQuestionGraph` from flat extracted fields. The graph preserves sections, field paths, conditional rules, repeat metadata, and stable field anchors.

### `getActiveApplicationFields(state)`

Return only fields whose field-level or graph-level conditions are currently satisfied.

### `planNextApplicationQuestions(state, limit?)`

Return `{ status, fieldIds, fields }` for active unanswered questions.

### `applyApplicationAnswers(state, answers, now?)`

Apply host-provided answers to a state object and recompute quality plus completion status.

### `proposeContextWrites(state)`

Build host-reviewable context proposals from source-backed or user-confirmed answers.

### `buildApplicationPacket(state, options?)`

Build a deterministic packet artifact. Packet status is `draft` when active required fields are missing or quality has blocking issues, otherwise `broker_ready`.

### `validateApplicationPacket(packet)`

Return the packet quality report with packet-level missing-answer validation.

## Prompt builders (advanced)

All application processing functions are also exported as prompt builders — they return prompt strings for use with any LLM.

Use the `@claritylabs/cl-sdk/application` subpath when a host runtime only needs application intake helpers. The root SDK export also includes extraction and PDF utilities, while this subpath keeps application state, question graph, context proposal, and packet logic isolated for browser-like/serverless runtimes such as Convex functions.

## Detection

### `APPLICATION_CLASSIFY_PROMPT`

A static prompt string for classifying whether a PDF is an insurance application form.

```typescript
import { APPLICATION_CLASSIFY_PROMPT } from "@claritylabs/cl-sdk";
```

**Expected response schema:**

```json
{
  "isApplication": boolean,
  "confidence": number,
  "applicationType": string | null
}
```

## Field extraction

### `buildFieldExtractionPrompt()`

Build a prompt to extract all fillable fields from an application PDF as structured data.

**Returns:** `string`

**Field types:** `text`, `numeric`, `currency`, `date`, `yes_no`, `table`, `declaration`

## Auto-fill

### `buildAutoFillPrompt(fields, orgContext)`

Build a prompt to match extracted fields against known business context for auto-filling.

| Parameter | Type | Description |
|-----------|------|-------------|
| `fields` | `{ id, label, fieldType, section }[]` | Extracted application fields |
| `orgContext` | `{ key, value, category }[]` | Organization context key-value pairs |

**Returns:** `string`

## Question batching

### `buildQuestionBatchPrompt(unfilledFields)`

Build a prompt to organize unfilled fields into topic-based batches for user interaction.

| Parameter | Type | Description |
|-----------|------|-------------|
| `unfilledFields` | `{ id, label?, text?, fieldType, section, required, condition? }[]` | Fields needing user input |

**Returns:** `string`

## Answer parsing

### `buildAnswerParsingPrompt(questions, emailBody)`

Build a prompt to parse free-text user replies into structured field values.

| Parameter | Type | Description |
|-----------|------|-------------|
| `questions` | `{ id, label?, text?, fieldType }[]` | Questions that were asked |
| `emailBody` | `string` | The user's reply text |

**Returns:** `string`

## Email generation

### `buildBatchEmailGenerationPrompt(batchFields, batchIndex, totalBatches, appTitle, totalFieldCount, filledFieldCount, previousBatchSummary?, companyName?)`

Build a prompt to generate an email asking users for missing application information.

| Parameter | Type | Description |
|-----------|------|-------------|
| `batchFields` | `{ id, label, fieldType, options?, condition? }[]` | Fields in this batch |
| `batchIndex` | `number` | Current batch number |
| `totalBatches` | `number` | Total number of batches |
| `appTitle` | `string \| undefined` | Application title |
| `totalFieldCount` | `number` | Total fields across all batches |
| `filledFieldCount` | `number` | Fields already filled |
| `previousBatchSummary` | `string` | Summary of previous batch (optional) |
| `companyName` | `string` | Company name for personalization (optional) |

**Returns:** `string`

## Reply classification

### `buildReplyIntentClassificationPrompt(questions, emailBody)`

Build a prompt to classify user reply intent (answering questions, asking for help, etc.).

| Parameter | Type | Description |
|-----------|------|-------------|
| `questions` | `{ id, label }[]` | Questions from the current batch |
| `emailBody` | `string` | The user's reply text |

**Returns:** `string`

## Field explanation

### `buildFieldExplanationPrompt(field, question, policyContext?)`

Build a prompt to generate a human-readable explanation of a complex insurance form field.

| Parameter | Type | Description |
|-----------|------|-------------|
| `field` | `{ id, label, fieldType, options? }` | The field to explain |
| `question` | `string` | The user's question about the field |
| `policyContext` | `string` | Optional policy context for more specific explanations |

**Returns:** `string`

## Confirmation

### `buildConfirmationSummaryPrompt(fields, applicationTitle)`

Build a prompt to generate a summary of all filled fields for user review before PDF generation.

| Parameter | Type | Description |
|-----------|------|-------------|
| `fields` | `{ id, label?, text?, section, fieldType, value? }[]` | All application fields |
| `applicationTitle` | `string` | Title of the application |

**Returns:** `string`

## PDF mapping

### `buildAcroFormMappingPrompt(extractedFields, acroFormFields)`

Build a prompt to map filled field values to AcroForm field names in a fillable PDF.

| Parameter | Type | Description |
|-----------|------|-------------|
| `extractedFields` | `{ id, label, value? }[]` | Extracted and filled fields |
| `acroFormFields` | `{ name, type, options? }[]` | AcroForm fields from the PDF |

**Returns:** `string`

### `buildFlatPdfMappingPrompt(extractedFields)`

Build a prompt to map filled field values to percentage-based page coordinates for flat PDF overlay.

| Parameter | Type | Description |
|-----------|------|-------------|
| `extractedFields` | `{ id, label, value, fieldType }[]` | Extracted and filled fields |

**Returns:** `string`

## Lookup fill

### `buildLookupFillPrompt(requests, targetFields, availableData)`

Build a prompt to fill fields by looking up values from existing insurance documents or data sources.

| Parameter | Type | Description |
|-----------|------|-------------|
| `requests` | `{ type, description, targetFieldIds }[]` | Lookup requests |
| `targetFields` | `{ id, label, fieldType }[]` | Target fields to fill |
| `availableData` | `string` | Available data to search through |

**Returns:** `string`
