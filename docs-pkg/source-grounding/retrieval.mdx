---
title: Source Retrieval
description: Searching source nodes and exact source spans deterministically
---

Source retrieval is intentionally small. Hosts can use the built-in lexical memory store, wrap an existing vector index, or combine source nodes with exact source spans. In v3, `searchSourceNodes` is the preferred path for document-backed Q&A.

## Interface

```typescript
interface SourceRetriever {
  searchSourceSpans(query: SourceRetrievalQuery): Promise<SourceRetrievalResult[]>;
  searchSourceNodes?(query: SourceRetrievalQuery): Promise<SourceNodeRetrievalResult[]>;
}

interface SourceNodeRetrievalResult {
  node: DocumentSourceNode;
  relevance: number;
  hierarchy: DocumentSourceNode[];
  spans: SourceSpan[];
}

interface SourceRetrievalQuery {
  question: string;
  documentIds?: string[];
  chunkIds?: string[];
  limit?: number;
  mode?: "graph_only" | "source_rag" | "long_context" | "hybrid";
  filters?: Record<string, string>;
}
```

`mode` is advisory. A host can interpret it as:

- `graph_only`: use structured relationships and exact metadata.
- `source_rag`: use source-node retrieval, then exact source spans.
- `long_context`: provide larger source windows when the provider supports them.
- `hybrid`: combine structured filters with lexical/vector retrieval.

`searchSourceNodes` should return the matched node, a bounded hierarchy packet, and exact spans. A good packet usually includes ancestors, the matched node, nearby siblings, selected children, and the source spans referenced by those nodes. This gives the reasoner enough structure to understand schedules and clauses without sending an entire policy.

## Memory store

```typescript
import { MemorySourceStore } from "@claritylabs/cl-sdk";

const store = new MemorySourceStore();
await store.addSourceSpans(spans);
await store.addSourceChunks(chunks);

const evidence = await store.searchSourceSpans({
  question: "What is the general liability per occurrence limit?",
  documentIds: ["policy-123"],
  filters: { sectionId: "declarations" },
  limit: 5,
});
```

The memory store uses lexical matching and deterministic tie-breaking. It is useful for tests, local development, and as a reference implementation.

For production source-tree retrieval, index `DocumentSourceNode.description` and keep `sourceSpanIds` on each node. When a node matches, hydrate its source spans from your source-span table so citations can open the exact PDF page and bounding boxes.

## Deterministic ordering

Use `orderSourceEvidence()` when combining evidence from multiple stores:

```typescript
import { orderSourceEvidence } from "@claritylabs/cl-sdk";

const ordered = orderSourceEvidence([...vectorResults, ...structuredResults]);
```

Evidence is ordered by descending relevance, then by a stable source identifier. That prevents equal-score retrieval from changing packet contents between runs.
