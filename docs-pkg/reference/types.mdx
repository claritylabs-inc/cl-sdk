## Provider callbacks

The SDK is provider-agnostic. Consumers wrap their preferred provider (Anthropic, OpenAI, etc.) into these plain callback functions. The SDK never imports or depends on any provider package.

### `GenerateText`

```typescript
type GenerateText = (params: {
  prompt: string;
  system?: string;
  maxTokens: number;
  providerOptions?: Record<string, unknown>;
}) => Promise<{ text: string; usage?: TokenUsage }>;
```

### `GenerateObject<T>`

```typescript
type GenerateObject<T = unknown> = (params: {
  prompt: string;
  system?: string;
  schema: ZodSchema<T>;
  maxTokens: number;
  providerOptions?: Record<string, unknown>;
}) => Promise<{ object: T; usage?: TokenUsage }>;
```

In the extraction pipeline, `providerOptions` can include:

- `pdfBase64` — the PDF to attach to the provider request
- `images` — rendered page images to attach to the provider request
- `sourceSpans` — source evidence relevant to the current model call
- `sourceChunks` — grouped source evidence windows derived from spans
- `attachments` — query/application attachment payloads when available

The coordinator uses the full PDF for classify, form inventory, and review. Page mapping and worker extractors use cached page-scoped PDFs, unless `convertPdfToImages` is enabled, in which case rendered image ranges are cached and passed through `providerOptions.images`.

### `EmbedText`

```typescript
type EmbedText = (text: string) => Promise<number[]>;
```

### `ConvertPdfToImagesFn`

```typescript
type ConvertPdfToImagesFn = (
  pdfBase64: string,
  startPage: number,
  endPage: number,
) => Promise<Array<{ imageBase64: string; mimeType: string }>>;
```

```typescript

```

### `LogFn`


## Extraction types

### `ExtractorConfig`

Configuration for `createExtractor()`. See [Extraction API](/docs/api/extraction) for full details.

```typescript
interface ExtractorConfig {
  generateText: GenerateText;
  generateObject: GenerateObject;
  convertPdfToImages?: ConvertPdfToImagesFn;
  concurrency?: number;           // Default: 2
  pageMapConcurrency?: number;    // Defaults to concurrency
  extractorConcurrency?: number;  // Defaults to concurrency
  formatConcurrency?: number;     // Defaults to concurrency
  maxReviewRounds?: number;       // Default: 2
  reviewMode?: "always" | "auto" | "skip";
  onTokenUsage?: (usage: TokenUsage) => void;
  onProgress?: (message: string) => void;
  log?: LogFn;
  providerOptions?: Record<string, unknown>;
  sourceStore?: SourceStore;
  qualityGate?: "off" | "warn" | "strict";
  modelCapabilities?: ModelCapabilities;
  modelBudgetConstraints?: Partial<Record<ModelTaskKind, ModelBudgetConstraint>>;
}
```

### `ExtractionResult`

```typescript
interface ExtractionResult {
  sourceTree?: DocumentSourceNode[];
  operationalProfile?: PolicyOperationalProfile;
  document: InsuranceDocument;
  chunks: DocumentChunk[];
  sourceSpans: SourceSpan[];
  sourceChunks: SourceChunk[];
  warnings?: string[];
  tokenUsage: TokenUsage;
  usageReporting: {
    modelCalls: number;
    callsWithUsage: number;
    callsMissingUsage: number;
  };
  performanceReport: PerformanceReport;
}
```

When source spans are provided, v3 extraction returns `sourceTree` and `operationalProfile`, and `chunks` is empty because broad structured chunks are no longer the primary evidence corpus. `document` remains a compatibility projection for hosts that still render `InsuranceDocument`.
