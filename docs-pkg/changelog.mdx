All notable changes to `@claritylabs/cl-sdk` are documented here.

## v1.0.3 -- Zod 4 Metadata Compatibility

Released 2026-05-09.

### Fixed

- Updated source span, source chunk, and case evidence metadata schemas to use the Zod 3/4-compatible two-argument `z.record(z.string(), z.string())` form.
- Added a regression test covering `SourceChunkSchema.parse()` with non-empty metadata such as `sourceKind` and `sectionId`.

## Unreleased

### Changed

- Extraction now accepts host-provided Docling documents as an optional pre-processed input, derives source spans from Docling provenance, and runs focused extraction over page-scoped Docling text without adding a Docling runtime dependency.
- Extraction now caches full-document provider options, page-scoped PDF ranges, rendered image ranges, and in-flight range work during a single run.
- Added separate `pageMapConcurrency`, `extractorConcurrency`, and `formatConcurrency` controls, plus bounded parallel formatting batches.
- Added `reviewMode` for choosing between automatic, always-on, and skipped LLM extraction review.
- Supplementary extraction can now run as narrow page-scoped tasks when page-map or form-inventory signals identify likely notice, regulatory, claims, or contact pages.

## v1.0.0 -- Source-Grounded Insurance Workflows

Released 2026-05-08.

This is the first stable CL SDK release line. The release promotes the SDK from an extraction-focused 0.x package into a source-grounded insurance workflow toolkit covering extraction, query, application processing, policy change endorsements, and reusable case state.

### Added

- **Source grounding** -- new `SourceSpan`, `SourceChunk`, `SourceSpanRef`, `SourceStore`, `SourceRetriever`, `MemorySourceStore`, source builders, source chunking, deterministic evidence ordering, and extraction source-grounding helpers.
- **Extraction source support** -- `extract(pdfBase64, documentId, { sourceSpans })` accepts caller-provided source spans, forwards page-filtered spans into focused extractors, derives `sourceChunks`, and can persist both spans and chunks through `sourceStore`.
- **Extraction quality and observability** -- extraction results now include `performanceReport`; source-aware quality gates detect weak grounding, missing source references, unresolved referential values, and placeholder-style extraction output.
- **Model budget resolution** -- `resolveModelBudget()` and model capability metadata let extraction, query, application, and PCE tasks choose safer per-task token budgets instead of relying on fixed prompt limits.
- **Query source retrieval** -- query agents can receive a `sourceRetriever` and retrieval mode hints so answers can combine document chunks, structured document lookup, conversation history, attachments, and source-span evidence.
- **Application source grounding** -- application processing accepts `sourceSpans`, forwards them through provider calls, and supports `replySourceSpanIds` for preserving provenance on user-supplied answers.
- **Policy change endorsements** -- new PCE workflow exports: `createPceAgent`, `collectPceEvidenceSources`, `validatePceItems`, `selectPceExecutionMode`, `buildPceSubmissionPacket`, `buildPceQualityReport`, `stablePolicyChangeItemId`, and PCE prompt builders.
- **Case workflow primitives** -- new reusable case schemas, evidence/citation validation, missing-info merge helpers, stable case IDs, proposal scoring, proposal evaluation, and packet artifact types.
- **Hybrid quality gates** -- extraction, query, application, and PCE workflows expose deterministic review reports so host products can separate draft, assisted, and automation-eligible flows.
- **Durable docs package update** -- docs now include v1 migration, source grounding, PCE, case workflow, API, storage, and quality documentation.

### Changed

- **Package versions move from 0.x to 1.0.0** for both `@claritylabs/cl-sdk` and `@claritylabs/cl-sdk-docs`.
- **Provider options are richer**. Hosts should preserve `providerOptions.sourceSpans` and `providerOptions.sourceChunks` when source evidence is available, in addition to existing PDF, image, and attachment fields.
- **Extraction results are broader**. Consumers should expect `sourceSpans`, `sourceChunks`, and `performanceReport` in addition to `document`, `chunks`, `tokenUsage`, `usageReporting`, and `reviewReport`.
- **Query and application configuration grew** with optional source retrievers, quality gates, model capabilities, and model budget constraints.
- **Docs navigation changed** to make source grounding, PCE, case workflows, and v1 migration first-class sections.

### Migration Notes

- Existing extraction, query, and application calls continue to work without source spans.
- For best v1 behavior, create source spans during ingest, pass them into extraction/application calls, and implement `SourceStore` or `SourceRetriever` beside your existing document and memory stores.
- Treat PCE submission packets as draft or assisted artifacts until a host-side human review and compliance flow approves carrier submission.
- Update documentation consumers to `@claritylabs/cl-sdk-docs@^1.0.0`.

## v0.11.0 — Declarations Promotion & Summary Generation

### Added

- **Declarations-to-schema promotion pass** — new Phase 7 in the extraction pipeline. After assembly, a promotion pass fills top-level typed fields from extracted declarations and coverage arrays. This bridges the gap between detailed extraction (100+ declaration fields, dozens of coverages) and the typed `InsuranceDocument` schema. Runs seven sub-passes with no model calls:
  - **Carrier field mapping** — `naicNumber` → `carrierNaicNumber`, `admittedStatus` → `carrierAdmittedStatus`, `amBestRating` → `carrierAmBestRating`; builds `insurer` sub-object
  - **Broker promotion** — declarations broker/agent/producer → `brokerAgency`, `brokerContactName`, `producer`
  - **Loss payee / mortgage holder promotion** — declarations → `lossPayees[]`, `mortgageHolders[]`
  - **Location promotion** — groups declaration location fields → `locations[]`
  - **Limits synthesis** — coverage names → `limits` schedule (GL, auto, umbrella, WC patterns)
  - **Deductible synthesis** — most common deductible → `deductibles.perOccurrence`; detects SIR and aggregate
  - **Premium fallback** — declarations premium → `premium` when extractor missed it
  - New file: `src/extraction/promote.ts`
- **Summary generation** — new Phase 8. Generates a 1–3 sentence AI summary of the assembled document. Appears as `summary` on the final `InsuranceDocument`. Gracefully skipped on failure.
  - New file: `src/prompts/coordinator/summarize.ts`
- **Broker extraction in `carrier_info`** — extractor now captures `brokerAgency`, `brokerContactName`, and `brokerLicenseNumber` alongside carrier data
- **Loss payee / mortgage holder extraction in `named_insured`** — extractor now captures `lossPayees[]` and `mortgageHolders[]` alongside insured data

### Fixed

- **Field name inconsistency** — carrier extractor output used short names (`naicNumber`, `admittedStatus`) that didn't match the `PolicyDocument` schema's prefixed names (`carrierNaicNumber`, `carrierAdmittedStatus`). The promotion pass now maps these correctly.

### Changed

- Pipeline now runs **10 phases** (was 8): Classify → Page Map → Plan → Extract → Review → Assemble → **Promote** → **Summarize** → Format → Chunk

## v0.7.x — Schema Compatibility & Resilience

### Added

- **`toStrictSchema()`** — auto-transforms Zod schemas before `generateObject` calls, converting `.optional()` to `.nullable()` for OpenAI strict structured output compatibility. Applied automatically in the pipeline.
- **`safeGenerateObject()`** — wraps `generateObject` with schema strictification, retry on schema validation errors, and optional fallback values when all retries are exhausted.
- **Pipeline checkpoints** — `createPipelineContext()` for save/resume at phase boundaries with `onCheckpointSave` callback.

### Fixed

- **`z.record()` removal** — replaced all `z.record()` usages in schemas with `z.array(z.object({ key, value }))` to avoid unsupported `propertyNames` in JSON Schema.
- **Transient error retry** — `withRetry()` now retries on grammar compilation timeouts, no-output-generated, overloaded, and 5xx errors (not just rate limits).
- **Graceful degradation** — extraction pipeline uses fallback values for classify/plan/review when all retries fail, allowing the pipeline to continue rather than crash.

## v0.5.0

### Added

- **Markdown formatting pass** — new Phase 6 in the extraction pipeline. After assembly, a formatting agent cleans up all content-bearing fields (sections, endorsements, exclusions, conditions, summary) to ensure proper markdown rendering. Fixes pipe tables missing separator rows, space-aligned tables, sub-items mixed into tables, and general whitespace issues.
  - New files: `src/extraction/formatter.ts`, `src/prompts/coordinator/format.ts`
  - Pipeline now runs 7 phases: Classify → Plan → Extract → Review → Assemble → Format → Chunk
  - Token usage from formatting is tracked and included in the total

## v0.1.0 — 2026-04-11

Version reset. All prior versions (1.x through 6.x) are no longer relevant and should not be used. The SDK will remain below 1.0.0 until official launch.

### Current Features

- Document classification (`classifyDocumentType`)
- PDF extraction pipeline (`extractFromPdf`, `extractQuoteFromPdf`)
- Provider-agnostic model configuration via Vercel AI SDK `LanguageModel` instances
- Composable agent prompt system (`buildAgentSystemPrompt`)
- Platform and intent model for multi-channel agents
- PDF operations (AcroForm fill, text overlay, page splitting)
- Comprehensive insurance type system (42 policy types, declarations, coverages)
- Rate-limit retry with exponential backoff
- Parallel chunk extraction with configurable concurrency
- Token usage tracking via `onTokenUsage` callback
- Context key mapping for policy-to-application auto-fill
- Native PDF file support with `convertPdfToImages` fallback
